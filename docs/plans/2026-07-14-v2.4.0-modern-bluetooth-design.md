# v2.4.0 "Modern Bluetooth" — Design

> **Post-validation corrections (2026-07-15):** hardware validation falsified
> two assumptions below. (1) The BRC1H advertises the local name "Daikin", not
> "BRC1H*" — the shipped manifest matches on the Madoka service UUID instead.
> (2) Proxy pairing is NOT transparent: the BRC1H requires authenticated MITM
> pairing that the client must initiate (pymadoka-ng ≥0.3.2 calls pair()) and
> that the stock bluetooth-proxy firmware cannot complete — see the README's
> Requirements section for the proxy YAML that was actually validated. The
> pymadoka dist was also renamed to pymadoka-ng (PyPI name + HA git-requirement
> reinstall workaround).

**Date:** 2026-07-14
**Scope:** coordinated release of `daikin_madoka` 2.4.0 (this repo) and the
`pymadoka` fork 0.3.0 (github.com/dasimon135/pymadoka).

## Goals

1. Make the direct-Bluetooth option work everywhere HA's Bluetooth stack works
   (local adapter **or** ESPHome Bluetooth proxies), with no Linux-only
   shell-outs.
2. Surface protocol data that pymadoka already parses but the integration never
   exposed (dual setpoints, device temperature limits, device info, RSSI).
3. Bring the integration up to current HA quality standards: coordinator,
   discovery, options flow, diagnostics, `has_entity_name`, fr translation.

## Non-goals

- Publishing the pymadoka fork to PyPI (packaging is prepared, upload is a
  manual follow-up — the dist name needs to be chosen since `pymadoka` is taken).
- HACS default submission (separate follow-up, needs the PyPI step first).
- Weekly schedule / other unreversed protocol features.

## pymadoka 0.3.0

### Packaging
- Replace `setup.py` with `pyproject.toml`.
- Core deps: `bleak>=0.21`, `bleak-retry-connector>=3.4`. `click`/`pyyaml`
  (CLI) and `asyncio-mqtt` (MQTT bridge) move to `[cli]` / `[mqtt]` extras so
  the HA install stays lean.
- Version 0.3.0, tagged; the integration pins the tag.

### Connection layer
- `bleak_retry_connector` becomes a hard import on the HA path (no silent
  fallback that masks failures).
- `force_device_disconnect()` (bluetoothctl subprocess) stays for the CLI but
  is never needed on the HA path; the integration stops calling it.
- Reconnect: keep in-library retry with capped backoff, but `start()` must
  return promptly when the device is out of range (bounded first attempt) so
  HA setup can raise `ConfigEntryNotReady` instead of hanging.
- Keep the `discover = BleakScanner.discover` legacy shim exported for CLI
  compat; the integration no longer uses library-side discovery.

### Tests + CI
- Unit tests for `Transport` chunk split/rebuild and `SetPointStatus`
  parse/serialize round-trips (pure python, no BLE needed).
- GitHub Actions: ruff + pytest.

## daikin_madoka 2.4.0

### Architecture
- New `coordinator.py`: `MadokaCoordinator(DataUpdateCoordinator)` per device.
  One poll (`controller.update()` + `refresh_status()`) shared by all
  entities; poll interval configurable via options flow (default 60 s).
- `__init__.py`: create controller + coordinator, `await
  async_config_entry_first_refresh()`; raise `ConfigEntryNotReady` when the
  device is unreachable. Drop the `force_device_disconnect` call and the
  `force_update` config semantics (option kept in stored entries for
  backward-compat, ignored).
- All entities become `CoordinatorEntity` subclasses with `has_entity_name`
  and `translation_key`. **unique_ids are unchanged** so no entity registry
  migration is needed.

### Discovery & config flow
- `manifest.json` gains a `bluetooth` matcher on `local_name: "BRC1H*"`
  (connectable). If a unit doesn't advertise that name, manual setup still
  works — the matcher is purely additive.
- Config flow v2:
  - `async_step_bluetooth` — confirm-only flow when HA discovers a unit.
  - `async_step_user` — dropdown of already-discovered connectable BRC1H
    devices plus a manual-MAC fallback field.
  - Adapter (`hci0`) and `force_update` fields removed from new entries
    (legacy entries keep working).
  - Options flow: poll interval.

### Entities
- `climate`: unchanged modes/fan; new — `min_temp`/`max_temp` read from the
  device's setpoint limits when available (fallback to consts); in AUTO mode
  with `range_enabled`, expose `TARGET_TEMPERATURE_RANGE` (separate
  cooling/heating setpoints, already writable through pymadoka).
- `sensor`: indoor + outdoor temperature (coordinator-backed) + new RSSI
  diagnostic sensor (from HA bluetooth advertisement data, disabled by
  default).
- `binary_sensor` (filter), `button` (reset filter), `number` (eye
  brightness): migrate to coordinator.
- Device registry: model / hw / sw versions from `controller.read_info()`.

### Quality
- `diagnostics.py` with MAC redaction.
- `strings.json` reworked + translations **en / es / fr**.
- README + CHANGELOG updated; version 2.4.0; requirement bumped to the
  pymadoka 0.3.0 tag.

## Risks

- **Discovery matcher name**: "BRC1H*" is the expected advertised name but is
  unverified on real hardware; harmless if wrong (manual flow remains).
- **Entity naming**: `has_entity_name` changes displayed names (not entity
  ids). Called out in the changelog.
- **AUTO-mode range**: only shown when the device reports `range_enabled`, so
  single-setpoint users see no change.
