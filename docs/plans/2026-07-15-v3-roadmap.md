# v3.0 Roadmap

**Status:** planned (scope agreed 2026-07-15). v2.4.0 ships first as-is.

## Pillar 1 — Madoka Card (flagship)

A Lovelace card that mirrors the physical BRC1H: round dial, glowing halo
ring (color follows state: heat/cool/off), centered temperature, fan-speed
segments, filter-alert badge, connection/RSSI indicator, integrated eye
brightness control.

**Architecture: embedded in the integration.** The card (LitElement, single
bundled JS) ships inside `custom_components/daikin_madoka/frontend/` and the
integration registers it automatically (static path + Lovelace resource) —
zero user configuration, one release cycle. The brand icon in `branding/`
sets the visual language (dark face, violet halo `#815AFF → #4069FF`).

## Pillar 2 — Guided setup & repairs

- Config flow performs the connection + pairing DURING setup: a progress step
  connects, initiates pairing, and tells the user "accept on the thermostat's
  display now" at the right moment, instead of creating a retry-looping entry.
- Repair issues: detect the looping `Insufficient authentication` signature
  and raise an actionable repair ("your Bluetooth proxy lacks pairing
  support") linking the README YAML. Detect never-advertising devices too.
- Reconfigure flow (change MAC without re-adding), diagnostic "Reconnect"
  button.

## Pillar 3 — Stats & adaptive polling

- Derived runtime statistics: heating/cooling time accumulation (from
  hvac_action), hours since last filter reset.
- Adaptive polling: fast poll right after a command or detected change,
  relaxed when idle overnight.
- Research task: check blafois' protocol notes for a native operating-hours
  query before deriving.

## Pillar 4 — Distribution & quality

- Publish `pymadoka-ng` on PyPI (kills the git-requirement reinstall trap
  for good), switch the manifest pin to a versioned requirement.
- Submit to HACS default.
- Test suite with `pytest-homeassistant-custom-component`; hassfest in CI;
  climb the HA quality scale (long-term option: core submission).

## Explicitly out of scope

- Vane/swing control: not exposed by the BRC1H BLE protocol.
- Weekly schedule read/write: not reverse-engineered; too risky.
